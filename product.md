# 通用软件可观测性框架产品文档（v2.0）

## 1. 产品概述

**产品名称**：通用软件可观测性框架（Universal Observability Framework, UOF）

**产品定位**：  
UOF 是一个面向 Linux 环境的通用可观测性框架，基于 **eBPF + OpenTelemetry + TSDB + Grafana**，为不同软件提供统一、低开销、可扩展的观测与诊断能力。

它不是某一个数据库、中间件或 Web 服务的专用监控工具，而是一个可复用的观测底座。数据库 DBA 只是一个典型示例场景，用于证明该框架能够帮助用户发现问题、分析问题并辅助解决问题。

**产品目标**：

1. 为 Linux 上运行的软件提供统一采集、统一建模、统一展示的观测能力。
2. 通过模板和插件机制，将底层探针能力复用到数据库、Web、中间件、缓存、消息队列等多个场景。
3. 降低新场景接入成本，让用户以较低代价获得可用的监控、告警和诊断能力。
4. 在观测之外，逐步增强异常检测、关联分析和根因定位能力。

**核心特点**：

1. **通用性**：支持任意支持 eBPF 的 Linux 系统，面向多种软件运行场景。
2. **低开销实时观测**：实时采集函数调用、内存分配、锁等待、系统事件、线程调度等信号。
3. **模板化落地**：通过模板和插件快速适配 Nginx、PostgreSQL、Redis、Kafka 等典型软件。
4. **统一语义模型**：将 Metrics、Trace、Log 和事件统一到同一套观测模型中，便于聚合和关联分析。
5. **安全可控**：Agent 和插件运行受控，eBPF 程序依赖内核验证器和平台治理机制保证安全性。

## 2. 目标用户与应用场景

### 2.1 目标用户

UOF 的目标用户包括但不限于：

* 平台工程师
* SRE / 运维工程师
* 数据库 DBA
* 中间件运维工程师
* 性能分析与故障定位工程师

### 2.2 首发示例场景：数据库 DBA

虽然 UOF 是一个通用框架，但数据库 DBA 是非常适合的首发示例场景，因为数据库问题通常同时涉及：

* 应用访问模式
* 数据库内部状态
* 操作系统资源
* 锁、IO、内存、线程和网络等多层信号

UOF 在该场景下可用于：

* 发现慢 SQL、锁等待、复制延迟、IO 抖动等问题
* 关联数据库指标与 OS / eBPF 事件
* 辅助 DBA 进行异常定位和性能瓶颈分析

### 2.3 其他可扩展场景

同一套框架可扩展到以下场景：

* Web 服务性能观测，如 Nginx、Apache
* 缓存系统观测，如 Redis
* 消息队列观测，如 Kafka、RabbitMQ
* 中间件与运行时观测，如 JVM、Go runtime

## 3. 产品价值

UOF 解决的不是“有没有数据”，而是“如何以统一方式快速建立场景化观测能力”。

**对平台侧的价值**：

* 提供统一的采集与治理底座，减少重复建设
* 为不同软件复用同一套观测模型、存储链路和可视化体系
* 通过模板化能力缩短接入和交付周期

**对一线使用者的价值**：

* 更快发现异常
* 更快定位问题
* 更低成本获得可用的 dashboard、告警和分析视图

**对 DBA 场景的价值示例**：

* 从“数据库慢了”快速定位到“锁等待、IO 抖动还是特定 SQL 模板异常”
* 将数据库内部指标与 OS 层行为进行关联分析
* 减少手工登录实例逐项排查的时间成本

## 4. 整体架构

### 4.1 高层架构

```text
+-----------------------------+
| Grafana / Alert / UI        |
| - 默认场景模板              |
| - 自定义 Dashboard          |
| - 告警与诊断入口            |
+-----------------------------+
              ^
              |
+-----------------------------+
| Template / Plugin Manager   |
| - 默认模板                  |
| - 用户插件                  |
| - 生命周期管理              |
| - 兼容性与治理              |
+-----------------------------+
              ^
              |
+-----------------------------+
| OTEL Collector / Pipeline   |
| - 接收 / 聚合 / 转换        |
| - 采样 / 压缩 / 路由        |
+-----------------------------+
              ^
              |
+-----------------------------+
| eBPF Agent                  |
| - 核心探针                  |
| - 用户态解析                |
| - 插件动态加载              |
+-----------------------------+
              ^
              |
+-----------------------------+
| Linux Target System         |
| - 数据库 / Web / 中间件     |
| - 任意支持 eBPF 的内核      |
+-----------------------------+
```

### 4.2 分层说明

UOF 建议按三层理解：

1. **观测底座层**  
   负责采集内核、进程、线程、函数、IO、网络等底层运行信号。

2. **统一观测模型层**  
   负责将不同来源的信号标准化为统一的实体、指标、事件与链路关系。

3. **场景交付层**  
   负责按软件类型和用户角色封装模板、插件、告警和 Dashboard。

## 5. 数据流与统一观测模型

### 5.1 数据流

1. **采集层**

   * eBPF 探针 hook 系统调用、函数入口/出口、内核事件、调度事件。
   * 通过 RingBuffer / perf buffer 输出到用户态 Agent。
   * Agent 对事件进行解析、聚合、采样和字段补全。

2. **处理层**

   * Agent 将数据转换为 Metrics / Trace / Log / Event。
   * OTEL Collector 统一接收并完成聚合、采样、压缩、路由。

3. **存储层**

   * 时序指标 → Prometheus / TSDB（TimescaleDB / InfluxDB）
   * Trace → Tempo / Jaeger
   * 日志与事件 → Loki / ElasticSearch

4. **消费层**

   * Grafana / 诊断 UI 进行展示、关联分析和告警输出。

### 5.2 统一观测模型

为保证通用性与可扩展性，UOF 需要定义统一观测模型，而不是仅输出原始探针数据。

**统一实体模型示例**：

* host
* container
* process
* thread
* service
* endpoint
* db_instance
* db_session
* query_template

**统一数据类型**：

* **Metric**：延迟、QPS、CPU、内存、IO、锁等待时长等
* **Trace**：请求链路、函数调用链、关键操作阶段耗时
* **Log / Event**：异常事件、告警事件、状态变化事件

**统一关联关系**：

* host -> process -> thread
* service -> endpoint -> request
* db_instance -> db_session -> query_template
* process / thread 与 syscall、IO、CPU、锁事件关联

该模型的目标是让不同插件产出的数据能够被统一聚合、对比和分析，而不是形成一个个孤立模板。

## 6. 功能模块

### 6.1 核心 Agent

**功能**：

* 采集函数调用、内存分配、系统调用、锁事件、线程调度、网络和磁盘 IO 行为
* 提供低开销、可采样、可限流的数据采集能力
* 支持多语言、多进程、多容器环境

**技术手段**：

* eBPF kprobe
* uprobe
* tracepoint
* perf event

### 6.2 OTEL Collector / Aggregation

**功能**：

* 统一接收 Agent 数据
* 完成 Metrics / Trace / Log / Event 转换与路由
* 支持批量发送、聚合、采样和多协议传输

**支持协议**：

* gRPC
* HTTP
* Kafka

### 6.3 Template / Plugin 系统

#### 模板能力

模板用于封装某一类软件或角色场景的默认交付物，包括：

* 推荐探针组合
* 指标定义
* Trace / Event 映射规则
* Dashboard
* 告警规则
* 场景说明与诊断路径

#### 插件开发接口

插件可扩展以下能力：

* 自定义 eBPF 探针
* 自定义 Metrics / Trace / Log / Event 输出
* 自定义实体映射规则
* 自定义 Dashboard 和告警模板

生命周期管理：

* `install()`
* `enable()`
* `disable()`
* `upgrade()`
* `uninstall()`

#### 插件治理能力

为避免插件体系失控，平台应提供：

* 版本兼容性声明
* 资源开销预算
* 沙箱与权限隔离
* 签名或来源校验
* 失败回滚与自动禁用机制

#### 加载流程

1. Agent 启动并初始化核心探针
2. Plugin Manager 加载默认模板
3. 加载用户插件与场景模板
4. 注册指标、Trace、事件和 Dashboard
5. Collector 完成接收、处理与转发
6. Grafana / UI 展示观测结果

### 6.4 可视化与诊断

* Grafana Dashboard 展示默认模板和自定义插件指标
* 支持 Trace 追踪、热力图、趋势图、表格分析
* 支持 Alert：延迟异常、热点函数、内存泄漏、锁等待、IO 抖动等
* 逐步增强异常检测、关联分析和根因候选输出能力

## 7. 典型模板示例：数据库 DBA 场景

数据库 DBA 不是产品边界，而是一个典型模板示例，用于说明 UOF 如何从通用框架落地到垂直场景。

### 7.1 目标

帮助 DBA 完成以下任务：

* 发现数据库性能异常
* 分析锁、IO、CPU、内存、连接和复制相关问题
* 将数据库现象与 OS 层行为进行关联
* 辅助输出定位路径和处置建议

### 7.2 默认观测项

以 PostgreSQL 或 MySQL 为例，可预置以下模板能力：

* 查询延迟分布
* 活跃会话数
* 慢 SQL Top N
* 锁等待与阻塞链
* Buffer / Cache 命中率
* WAL / Redo 压力
* 磁盘 IO 延迟与吞吐
* 复制延迟

### 7.3 诊断路径示例

当 DBA 发现数据库延迟升高时，模板可提供一条标准分析链路：

1. 查看实例级延迟、QPS、连接数是否异常
2. 下钻到慢 SQL 模板、活跃会话或锁等待情况
3. 关联对应进程的 CPU、内存、syscall、磁盘 IO 和调度事件
4. 输出可能原因，如锁竞争、磁盘抖动、热点查询、复制阻塞等

### 7.4 模板价值

数据库模板的意义不只是服务 DBA，更重要的是验证 UOF 的模板化方法：

* 同一套底座可以被封装成可直接使用的场景产品
* 底层能力与上层诊断视图可以解耦复用
* 数据库场景验证通过后，可复制到 Web、中间件和缓存场景

## 8. 产品边界

| 类别 | 边界说明 |
| --- | --- |
| 支持系统 | Linux 内核支持 eBPF 的版本，建议 >= 4.18 |
| 不支持系统 | Windows、macOS、以及不支持 eBPF 的内核环境 |
| 数据精度 | 重点采集热点函数、关键对象和关键行为，不做全量对象图 |
| 资源开销 | Agent 目标为低开销采集，但不保证硬实时 |
| 权限 | Agent 需要 BPF 相关权限，插件在受控环境中运行 |
| 可扩展性 | 默认模板覆盖常见软件，其他场景通过模板和插件扩展 |
| 产品范围 | 以性能分析、异常定位和诊断辅助为主，不做业务报表平台 |

## 9. 技术挑战与注意事项

1. **内核兼容性**：不同内核版本支持的 eBPF 特性差异较大，需要探针 fallback 策略。
2. **性能开销控制**：大量 hook 会影响业务，需要采样、聚合、限流和动态降级能力。
3. **高并发数据量治理**：高流量场景需要批量发送、压缩、背压和丢弃策略。
4. **统一语义建模难度**：不同软件的数据结构和诊断口径不同，统一模型设计复杂。
5. **跨层关联分析复杂**：需要解决时间戳对齐、上下文传播和实体映射问题。
6. **插件稳定性治理**：需要解决热加载、升级、隔离、回滚和兼容性管理问题。
7. **安全性**：要防止插件破坏目标系统或泄漏敏感数据。

## 10. MVP 范围与版本路线

### 10.1 MVP 范围

首版建议重点验证“通用框架 + 场景模板”方法，而不是一次性覆盖所有类型软件。

**MVP 建议包含**：

* 核心 Agent 与基础探针能力
* OTEL Collector 接入链路
* 统一 Metric / Event / Trace 基础 schema
* 模板与插件管理最小能力
* 一个完整的数据库 DBA 模板示例
* 一到两个非数据库模板示例，如 Nginx、Redis

**MVP 暂不强调**：

* 完整插件市场
* 全量自动根因分析
* 大规模多租户治理能力
* 复杂商业报表与管理后台

### 10.2 版本路线

**v1.0**

* 完成观测底座
* 提供数据库 DBA 模板示例
* 支持基础 Dashboard、告警和诊断路径

**v1.5**

* 补充模板脚手架和插件治理能力
* 增加 Redis、Kafka、Nginx 等模板
* 加强跨层关联分析

**v2.0**

* 增强异常检测和根因候选能力
* 支持更成熟的模板生态和可视化诊断工作台
* 提供更完整的企业级治理能力

## 11. 产品化部署建议

* **Agent**：以 DaemonSet 部署在 Linux 节点，可用于裸机或 K8s 集群
* **Collector**：集中部署，统一接收 Agent 数据
* **存储**：Prometheus + Tempo + Loki，必要时扩展 TSDB
* **可视化**：Grafana 作为默认展现层，后续可扩展专用诊断 UI
* **插件管理**：提供 CLI 或 Web 界面管理模板、版本和用户插件

---

## 12. 总结

UOF 的本质不是单一场景下的监控工具，而是一个可面向多软件、多角色复用的通用可观测框架。

它的核心竞争力在于：

1. **统一底座**：以 eBPF 为基础，统一采集 Linux 运行时关键行为。
2. **统一模型**：将指标、链路、日志和事件标准化，便于跨场景复用。
3. **模板化落地**：通过模板和插件快速生成可用的场景能力。
4. **场景可扩展**：数据库 DBA 可以作为示例场景先落地，再逐步复制到更多软件类型。
5. **从观测走向诊断**：在采集和展示之上，逐步增强异常检测、关联分析和问题定位能力。
